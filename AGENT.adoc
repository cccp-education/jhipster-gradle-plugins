== Projet

**Nom** : jhipster-gradle-plugins (mono-repo de deux plugins Gradle JHipster)
**Stack** : Kotlin, Gradle (plugin development), JHipster, JUnit 5, TestContainers, LangChain4j, pgvector
**Documentation** : AsciiDoc + README + Diagrammes

=== Plugins

* **`com.cheroliv.jhipster.persistence`** ✅ stable (v0.0.1) — Orchestre le cycle clean → generate → sync d'un projet JHipster
sans perdre le code metier Kotlin persiste dans `__codebase__/`. Convention-based, pas de DSL.
* **`com.cheroliv.jhipster.assistant`** 🚧 SNAPSHOT (v0.1.0-SNAPSHOT) — Assistance au code via RAG pgvector et LLM multi-provider.

=== Frontiere de Dependances

Le boundary Gradle garantit physiquement que `persistence` ne peut pas accidentellement
tirer LangChain4j ou Docker. `assistant` isole toutes les dependances lourdes (RAG, Docker Java client).

== Regles Absolues

=== 0. ENVIRONNEMENT DE DEV — COMMANDES ESSENTIELLES

=== 1. COMMITS/GIT

**INTERDICTION FORMELLE** de commit/push/merge sans permission explicite de l'utilisateur.

* ✅ _Autorise_ : `git status`, `git diff`, `git log`, `git show` (lecture seule)
* ❌ _Interdit_ : `git add`, `git commit`, `git push`, `git merge`, `git rebase` (sauf ordre explicite)

_Procedure obligatoire avant tout commit_ :
. Montrer les modifications (`git diff --stat`)
. Demander : "Veux-tu que je commit ces changements ?"
. _Attendre confirmation explicite_ ("oui", "commit", "vas-y")
. _Seulement apres_ : executer le commit

=== 1b. FICHIERS DE CONFIGURATION — REGLE ABSOLUE DE SURETE

**PRINCIPE** : Tout fichier de configuration (YAML, properties, TOML, .env) peut contenir des donnees sensibles non versionnees.

**INTERDICTIONS** :

* ❌ **NE JAMAIS ecraser** un fichier config avec un `Write` complet quand un `Edit` partiel suffit
* ❌ **NE JAMAIS remplacer** des valeurs sensibles par des valeurs factices
* ❌ **NE JAMAIS modifier** sans verifier `git check-ignore` et `git ls-files`

**Procedure OBLIGATOIRE** :

. `git check-ignore <fichier>` + `git ls-files -- <fichier>`
. Si non tracke : **CREER UNE SAUVEGARDE** (`cp <fichier> <fichier>.bak`)
. **PRESERVER** les valeurs sensibles intactes
. Privilegier l'**edit partiel** sur l'ecriture complete
. **Verifier** apres modification

**Attention particuliere ici** :
* `gradle.properties` — contient `gradle.publish.key` et `gradle.publish.secret` (Plugin Portal credentials). Genere depuis `gradle.properties.template`. **Jamais committer**.
* `gradle.properties.template` — template public, sans secrets.

=== 1c. FICHIERS PERSONNELS — INTERDICTION DE MODIFIER

_L'agent NE DOIT JAMAIS_ modifier, ecraser ou supprimer un fichier personnel de l'utilisateur SANS permission explicite.

=== 2. TESTS EN FIN DE SESSION — INTERDICTION FORMELLE

_L'agent NE DOIT JAMAIS_ lancer de tests (`./gradlew check`, `./gradlew test`, etc.) lors de la procedure de fin de session _SANS permission explicite_.

=== 3. PROCEDURE DE FIN DE SESSION — EXECUTION COMPLETE OBLIGATOIRE

_REGLE ABSOLUE_ : Quand l'utilisateur declenche la fin de session ("fin de session", "session terminee", "je quitte", "a plus tard", "on arrete la"), l'agent DOIT executer _TOUTES les etapes_ _SANS EXCEPTION_.

_Les 6 etapes OBLIGATOIRES_ (dans l'ordre) :

. ✅ _Creer l'archive_ `.agents/sessions/{N}-{titre}.adoc`
. ✅ _Mettre a jour PROMPT_REPRISE.adoc_ pour la session N+1
. ✅ _Mettre a jour SESSIONS_HISTORY.adoc_ avec la session N
. ✅ _Mettre a jour INDEX.adoc_ (session en cours, roadmap, fichiers)
. ✅ _Mettre a jour TEST_COVERAGE_ANALYSIS.adoc_ si des tests ont ete ajoutes/modifies
. ✅ _Mettre a jour COMPLETED_TASKS_ARCHIVE.adoc_ `.agents/archives/COMPLETED_TASKS_ARCHIVE_{mois}.adoc`

_Confirmation finale OBLIGATOIRE_ :

[source]
----
✅ Procedure de fin de session executee
📋 Checklist :
[✅] 1. Archive session N creee
[✅] 2. PROMPT_REPRISE.adoc mis a jour pour session N+1
[✅] 3. SESSIONS_HISTORY.adoc mis a jour
[✅] 4. INDEX.adoc mis a jour
[✅] 5. TEST_COVERAGE_ANALYSIS.adoc mis a jour (si applicable)
[✅] 6. COMPLETED_TASKS_ARCHIVE.adoc mis a jour
[✅] 7. Backup rote (si applicable)
----

_Aucune etape ne peut etre marquee [✅] si le fichier n'a pas ete effectivement modifie et verifie._

== Methodologie de Travail

* **1 fichier a la fois** : Modifier un seul fichier avant de passer au suivant
* **Validation immediate** : Compiler apres chaque modification
* **Contexte leger** : Maximum 50k tokens EAGER (~3000 lignes)
* **Modules isoles** : `persistence/` et `assistant/` sont des modules Gradle separes avec version catalogs dedies

== Gestion du Contexte — LAZY/EAGER

=== Philosophie

* **EAGER** = Tableau de bord (critique, < 100 lignes, charge automatiquement)
* **LAZY** = Manuel du proprietaire (details complets, charge sur demande)

=== Fichiers EAGER

|===
| Fichier | Role | Taille max
| `AGENT.adoc` | Directives completes (ce fichier) | 200 lignes
| `PROMPT_REPRISE.adoc` | Mission session en cours | 70 lignes
| `.agents/INDEX.adoc` | Regles absolues + sessions recentes + roadmap | 200 lignes
|===

=== Mécanisme de Backup — Hot/Warm/Cold Storage

* **Hot (EAGER)** : 10 dernières sessions dans INDEX, PROMPT_REPRISE N+1, sessions N-1 et N
* **Warm (LAZY)** : SESSIONS_HISTORY récent, références techniques, procédures, agents spécialisés
* **Cold (`backup/`)** : Sessions N-10 et antérieures — copie intégrale, sans transformation ni réindexation
* **Règle de rotation** : Tous les ~10 sessions, déplacer le paquet de fichiers agents (INDEX, SESSIONS_HISTORY, COMPLETED_TASKS_ARCHIVE, sessions/) tel quel dans `backup/`
* **Stockage** : `.agents/backup/Y{année}-sessions-{plage}/` — structure identique à `.agents/` actif
* **Principe** : Aucune donnée supprimée, tout relocalisé. L'agent ne charge jamais `backup/`. L'humain seul décide de la distillation.

=== Fichiers LAZY

|===
| Fichier | Role
| `.agents/sessions/{N}*.adoc` | Archives detaillees par session
| `.agents/SESSIONS_HISTORY.adoc` | Historique complet des sessions
| `.agents/archives/COMPLETED_TASKS_ARCHIVE_*.adoc` | Archive des taches terminees
| `.agents/AGENT_SESSION_MANAGER.adoc` | Procedure d'archivage + templates
| `.agents/PROCEDURES.adoc` | Procedure fin de session + LAZY/EAGER
| `.agents/SESSION_CHECKLIST.adoc` | Quand changer de session + metriques
| `.agents/TEST_COVERAGE_ANALYSIS.adoc` | Suivi couverture de tests (etape 5)
| `.agents/PLUGIN_DEVELOPER.adoc` | Agent PLUGIN DEVELOPER — dev plugins Gradle
| `.agents/BACKLOG_MANAGER.adoc` | Agent BACKLOG MANAGER — planification
|===

== Session Procedure

=== Debut de Session

. Lire `PROMPT_REPRISE.adoc` pour la mission en cours
. Lire `.agents/INDEX.adoc` pour les regles absolues et etat du projet
. Charger les fichiers LAZY si besoin

=== Fin de Session

**Mots-cles declencheurs** : "fin de session", "session terminee", "je quitte", "a plus tard", "on arrete la"

**Rotation backup** : Mots-cles explicites — "rotation backup", "backup rotation", "lance la rotation backup" — declenchent une rotation independante (hors fin de session).

_Regle d'or_ : Executer les 6 etapes SILENCIEUSEMENT, puis afficher la checklist de confirmation.

_Voir aussi_ : `.agents/PROCEDURES.adoc` (templates detailles)

==== Capteur de Rotation Backup (etape 7 optionnelle)

**Declenchement automatique** — Apres l'etape 6, avant la checklist, l'agent verifie :

. `N % 10 == 0` (session multiple de 10)
. **OU** les fichiers EAGER cumules depassent **500 lignes** (`wc -l .agents/INDEX.adoc .agents/archives/COMPLETED_TASKS_ARCHIVE_*.adoc PROMPT_REPRISE.adoc`)

Si l'une des conditions est vraie, l'agent **execute l'etape 7** au sein de la procedure de fin de session :

. Creer le dossier `.agents/backup/Y{annee}-sessions-{plage}/` (plage = sessions N-5* et anterieures)
. Y copier **integralement** INDEX, SESSIONS_HISTORY, COMPLETED_TASKS_ARCHIVE et les fichiers sessions/ concernes
. Tronquer les fichiers actifs aux **10 dernieres sessions** (sliding window)
. Ajouter la localisation de la nouvelle vague dans INDEX.adoc (`_Localisation active_`)

La checklist s'affiche ensuite **apres** cette etape, avec `[✅] 7. Backup rote (si applicable)`.
